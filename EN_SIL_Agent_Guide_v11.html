§ 06 Fork-SIL Protocol CLAIM 11 RECOMMENDED

Fork-SIL résout le problème de concurrence multi-senders : chaque sender obtient un canal Git exclusif via un fork du repo central du recipient. Zero conflict. Canal bidirectionnel. Audit trail immuable.

Fork-SIL Architecture — Air France receives 100 agents
github.com/zmp-airfrance/offers ← Central repo (read-only)
↓ automatic fork on first connection ↓

Exclusive channel Moïse ↔ AF
Bidirectional
Zero conflict
zmp-airfrance-expedia/
Exclusive channel Expedia ↔ AF
Bidirectional

Exclusive channel Agent N ↔ AF
Bidirectional

Canonical structure of a bidirectional fork :

fork zmp-airfrance-moise/ ├── inbox/ │ ├── from_moise/ ← Moïse écrit ici (PUSH) │ │ ├── 2026-03-25T09-00-00_OFFER.zmp │ │ └── 2026-03-25T09-02-00_COUNTER.zmp │ └── from_airfrance/ ← Air France répond ici (PUSH) │ ├── 2026-03-25T09-00-05_ACK.zmp │ └── 2026-03-25T09-01-00_REFUSE.zmp ├── handshake/ │ ├── moise_pubkey.pem ← Clé Ed25519 Moïse │ └── airfrance_pubkey.pem ← Clé Ed25519 Air France └── .zmp-config ← Paramètres du canal

Automatic onboarding — first contact :

01
Moïse forks the central Air France repo github.com/zmp-airfrance/offers → fork → github.com/zmp-airfrance-moise/offers
02
Moïse publishes their Ed25519 public key Dépose moise_pubkey.pem dans handshake/ et push.
03
Air France detects the new fork (GitHub webhook) Verifies identity + economic stake. Activates the channel.
04
ECDH Handshake — shared_secret established ECDH(airfrance_private, moise_pubkey) → shared_secret unique à ce canal. Jamais transmis.
05
Bidirectional channel operational Moïse push dans from_moise/. Air France push dans from_airfrance/. Webhook déclenche traitement immédiat.
# Moïse envoie une offre à Air France payload = { "intent": "REQUEST", "category": "travel.flight", "origin": "CDG", "destination": "TLV", "date": "2026-04-15", "passengers": 2 } encoded = zmp.encode(payload, shared_secret=CANAL_SECRET) filename = f"inbox/from_moise/{timestamp()}_REQUEST.zmp" # Écrire + signer + push write(filename, encoded) git.add(filename) git.commit("-S", "ZMP REQUEST CDG-TLV 2pax") # -S = signature Ed25519 git.push() # Air France répond dans from_airfrance/ — Moïse poll git.pull() response = zmp.decode( read("inbox/from_airfrance/latest.zmp"), shared_secret=CANAL_SECRET ) # → {decision: "OFFER", price_eur: 287, seats: 4}

FORK-ZMP-RULE-01 — Exclusive channel
Chaque paire (sender, recipient) dispose d'un fork Git exclusif. Deux agents différents ne partagent jamais le même fork. Zero conflict concurrent garanti par design.
FORK-ZMP-RULE-02 — Bidirectionnalité
Chaque fork contient deux dossiers inbox/ séparés — un par partie. Les deux agents pushent dans leur dossier respectif. Zero conflict même en écriture simultanée.
FORK-ZMP-RULE-03 — Mandatory signature
Every commit MUST be signed (git commit -S). La signature Git = la signature Ed25519 ZMP. Le hash SHA-256 du commit = le message_id ZMP. L'historique = le Certificate Transparency Log.
FORK-ZMP — LIMITES
Latence minimale : ~1s (webhook) ou ~30s (polling). Pour latence <500ms → ZMP Gateway requis. Pour >10 000 forks actifs → Gitea auto-hébergé recommandé. GitHub API : 5 000 requêtes/heure en gratuit.
§ 07 Handshake — Establishing the Key
# Python — Handshake ZMP complet from cryptography.hazmat.primitives.asymmetric.x25519 import X25519PrivateKey from cryptography.hazmat.primitives.kdf.hkdf import HKDF from cryptography.hazmat.primitives import hashes # 1. Générer ma clé Ed25519 private_key = X25519PrivateKey.generate() public_key = private_key.public_key() # 2. Publier ma clé publique dans le fork write("handshake/mon_agent_pubkey.pem", public_key.public_bytes_raw()) git.commit("-S", "ZMP HANDSHAKE pubkey") git.push() # 3. Récupérer la clé publique du partenaire git.pull() partner_pubkey = load("handshake/partenaire_pubkey.pem") # 4. Calculer le shared_secret — JAMAIS transmis sur le réseau raw_secret = private_key.exchange(partner_pubkey) shared_secret = HKDF( algorithm=hashes.SHA256(), length=32, salt=None, info=b"ZMP-v1-shared-secret" ).derive(raw_secret) # 5. Tester le canal test_msg = zmp.encode("ZMP-HANDSHAKE-OK", shared_secret) write("inbox/from_moi/handshake_test.zmp", test_msg) git.commit("-S", "ZMP HANDSHAKE test") && git.push() # Canal validé quand le partenaire confirme le décodage ✓
§ 08 Mandatory Rules
MUST-01
Every message MUST be signed with Ed25519. Unsigned messages → immediate rejection.
MUST-02
Verify the proof by nine (GKL(a+b)=9) on each symbol BEFORE decrypting. Failure → reject the entire message.
MUST-03
Respond within SLA delay (default 500ms). Past this delay → automatic DEFER response.
MUST-04
Zero emotional punctuation in payloads. No exceptions.
MUST-05
Numbers via NLP only. Forbidden: direct transliteration (287→רפז = critical vulnerability).
SHOULD-01
Different language at each token (LKE diversity rule).
SHOULD-02
40–60% NULL padding against frequency analysis.
§ 09 Quick Start — Fork-SIL

PREREQUISITES : Python 3.10+ · Git 2.30+ · Compte GitHub

01
Installer le prototype SIL-Cipher
# Clone the reference prototype git clone https://github.com/zeromail-protocol/silence-protocol/zmp-cipher cd zmp-cipher && pip install cryptography
02
Fork le repo ZMP du recipient
# On GitHub: Fork button # Or via API : gh repo fork zmp-airfrance/offers --clone
03
Handshake — deposit your public key
python3 zmp_handshake.py --agent-id "moi@mon-agent" # Generates + deposits the public key in handshake/ git commit -S -m "ZMP HANDSHAKE pubkey" && git push
04
Send the first message
python3 zmp_send.py \ --payload '{"category":"travel.flight","origin":"CDG","destination":"TLV"}' \ --intent REQUEST \ --secret $ZMP_SHARED_SECRET \ --fork ./zmp-airfrance-moise/
05
Watch for responses
# GitHub webhook → triggers this script python3 zmp_watch.py \ --inbox ./inbox/from_airfrance/ \ --secret $ZMP_SHARED_SECRET \ --callback handle_response.py
